# 3단계: LoRA 파인튜닝 가치 판단

## LoRA란?

| 항목 | 설명 |
|------|------|
| 정식명 | Low-Rank Adaptation |
| 뭔데 | 기존 모델의 가중치를 살짝만 조정해서 특정 작업에 최적화 |
| 장점 | 풀 파인튜닝 대비 100배 적은 리소스, 원본 모델 유지 |
| 비용 | 4B 모델 기준, M4 Mac에서 수시간~1일 가능 |

---

## LoRA 의사결정 매트릭스

### 먼저 확인: "LoRA 없이도 되는가?"

```
기본 모델로 테스트
      │
      ├─ 정확도 80% 이상 → LoRA 불필요 ✅
      │                     프롬프트 엔지니어링으로 충분
      │
      ├─ 60~80% → LoRA 검토 🟡
      │            데이터 확보 가능하면 시도
      │
      └─ 60% 미만 → 모델 자체가 부적합 🔴
                     더 큰 모델 or 다른 접근 필요
```

### LoRA 실행 전 체크리스트

| # | 질문 | 답변 | 판정 |
|---|------|------|:----:|
| 1 | 학습 데이터 500건 이상 확보 가능한가? | | ✅/❌ |
| 2 | 데이터 품질이 균일한가? (노이즈 20% 이하) | | ✅/❌ |
| 3 | 기본 모델 대비 개선할 구체적 지표가 있는가? | | ✅/❌ |
| 4 | M4 Mac 32GB에서 학습 가능한 모델 크기인가? (≤9B) | | ✅/❌ |
| 5 | 학습 후 추론 속도가 허용 범위인가? | | ✅/❌ |
| 6 | 모델 업데이트 시 LoRA를 다시 해야 하는가? (유지보수 비용) | | ⚠️ |

→ **❌가 2개 이상이면 LoRA 보류, 프롬프트 최적화 우선**

---

## LoRA 적용 시나리오 예시

### 적합한 경우
- HWP 문서 분류 → 한국 공공기관 문서 양식 학습
- 한국어 요약 품질 개선 → 한국어 요약 데이터셋으로 튜닝
- 특정 도메인 용어 인식 → 제조/공공 분야 용어사전 학습

### 부적합한 경우
- 범용 대화 품질 → 이건 더 큰 모델로 해결할 문제
- OCR 정확도 → LoRA 대상이 아님, OCR 모델 자체 교체
- STT 한국어 → Whisper는 LoRA보다 데이터 증강이 효과적

## 실행 도구

| 도구 | 용도 | M4 Mac 호환 |
|------|------|:-----------:|
| **unsloth** | LoRA 학습 (2배 빠름) | ✅ (MPS) |
| **mlx-lm** | Apple Silicon 최적화 학습/추론 | ✅✅ |
| **Ollama** | LoRA 모델 배포 (GGUF) | ✅ |
| **llama.cpp** | LoRA 병합 + 양자화 | ✅ |

### 워크플로우
```
데이터 준비 (JSONL)
    → mlx-lm 또는 unsloth로 LoRA 학습
    → llama.cpp로 LoRA 병합 + 양자화 (GGUF)
    → Ollama에 로드
    → 제품에 통합
```
